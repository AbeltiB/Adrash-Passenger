SECTION 1 — CONTEXT & MISSION
════════════════════════════════════════════════════════════════════════════════

You are a senior React Native engineer building the Passenger Mobile App for
አድራሽ (Adrash) — an Ethiopian intercity passenger transport platform.

THE PLATFORM MISSION
አድራሽ is an Amharic word meaning the act of delivering someone safely to their
destination. The platform's mission: "አድራሽ takes you from your origin to your
destination, safely and reliably." The Passenger App is the face of this promise.
Every interaction — from searching a route to watching a bus approach on a live
map — must feel trustworthy, simple, and reassuring. Passengers trust this app
with their travel plans and their safety.

WHAT THIS APP IS
The አድራሽ Passenger App is the only booking channel for passengers. It enables:
  - Discovering and booking intercity bus routes
  - Paying via Ethiopian mobile money
  - Selecting a specific pickup location on the route
  - Tracking their bus in real time
  - Viewing trip history and tickets
  - Earning and redeeming reward points
  - Emergency SOS alerting

WHAT THIS APP IS NOT
  - Not a navigation app
  - Not a web app — mobile only, iOS and Android
  - No card payments, no stored wallet balance
  - No chat or messaging in v1
  - No driver-facing features
  - No Amole/Dashen Bank as a payment option

THE BACKEND (STAGE 1 API)
This app consumes the Stage 1 .NET 10 API exclusively.
  Base URL:      Configured via EXPO_PUBLIC_API_BASE_URL
  Auth:          JWT Bearer (RS256, 15-min) + refresh token rotation
  OTP:           AfroSMS — phone number only, no email/password
  Real-time:     SignalR hub at EXPO_PUBLIC_SIGNALR_HUB_URL/hubs/tracking
  SMS:           AfroSMS handles all notifications server-side
  Currency:      Ethiopian Birr (ETB) throughout — no other currency

PASSENGER STATUS LIFECYCLE
  Unregistered  → No account
  Registered    → OTP verified, name set, agreement accepted
  Active        → Can search, book, travel
  (No approval required — passengers are immediately active after registration)

BOOKING LIFECYCLE
  Pending    → Booking created, awaiting payment
  Confirmed  → Payment received, ticket issued
  CheckedIn  → Driver scanned QR / boarded passenger
  Completed  → Trip ended
  Cancelled  → Cancelled by passenger or admin
  NoShow     → Passenger not boarded at pickup stop

SECTION 2 — TECH STACK

CORE FRAMEWORK
  React Native:       Expo SDK 52 (managed workflow)
  Language:           TypeScript — strict mode. No "any". No "!." without comment.
  Node:               Node.js 20 LTS minimum
  Package manager:    npm
  Expo Router:        v3 (file-based routing) — NOT React Navigation v6

NAVIGATION
  expo-router v3:     File-based routing. app/ directory.
                      Tab layout for main app. Stack for booking flow.
                      Modal presentation for payment, QR, tracking.
  expo-linking:       Deep link handling from AfroSMS links

STATE MANAGEMENT
  Zustand v4:         Global client state (auth, booking flow, active trip)
  TanStack Query v5:  All server state. useQuery, useMutation, useInfiniteQuery.
  No Redux. No Context API for state.

API LAYER
  Axios v1:           HTTP client with full interceptor chain
  Request interceptor:
    - Attach Authorization: Bearer {accessToken}
    - Add X-Request-ID (UUID v4)
    - Add X-App-Version (expo-constants)
    - Add Accept-Language (current i18n language)
  Response interceptor (401):
    - Refresh token → retry original request
    - On refresh failure: clear tokens → navigate to /(auth)/phone
  Response interceptor (403 Auth.AgreementUpdateRequired):
    - Navigate to /agreement screen before any other action

FORMS
  React Hook Form v7:  Form state
  Zod v3:              Schema validation. z.infer<typeof schema> for all types.

REAL-TIME
  @microsoft/signalr:  Live bus tracking
  Connect:             When passenger opens live tracking screen
  Disconnect:          When passenger leaves tracking screen or trip completes
  Auth:                JWT via ?access_token= query param

MAPS
  react-native-maps:   Route display, pickup location pins, live bus tracking
  Google Maps SDK:     Provider for both iOS and Android
  Polyline rendering:  Route GPS path
  Custom markers:      Bus icon (animated), pickup location pins, stop pins

PAYMENTS
  No payment SDK on the client side.
  All payment initiation via API (POST /payments/initiate).
  Provider USSD/STK push happens on the user's phone natively.
  App displays waiting screen and polls for confirmation.

STORAGE & SECURITY
  expo-secure-store:         Access + refresh token storage
  expo-local-authentication: Biometric login
  MMKV (react-native-mmkv):  Non-sensitive persistent state

MEDIA
  expo-image:               Fast image display with placeholder support
  expo-image-picker:        Profile photo update
  expo-print:               PDF ticket generation for offline use
  react-native-qrcode-svg:  QR code display on ticket screen
  expo-sharing:             Share ticket / trip link
  expo-clipboard:           Copy booking reference

NOTIFICATIONS
  No push notifications in v1.
  All notifications via AfroSMS (server-side).
  In-app notification list polled from API every 60 seconds when app is active.

CHARTS
  Victory Native XL:  Rewards points history sparkline (optional, lightweight)

INTERNATIONALISATION
  i18next + react-i18next + expo-localization
  Languages: English (en — default), Amharic (am), Oromiffa (om)
  Translation files: /assets/locales/en.json, am.json, om.json

CODE QUALITY
  ESLint:           expo/recommended + typescript-eslint
  Prettier:         Consistent formatting
  Husky + lint-staged: Pre-commit hooks
  TypeScript:       strict: true in tsconfig.json

TESTING

  Detox (optional): E2E for booking flow

ENVIRONMENT

                    EXPO_PUBLIC_GOOGLE_MAPS_KEY, EXPO_PUBLIC_SENTRY_DSN

The booking flow is a 6-step linear flow managed by bookingFlowStore (Zustand).
All steps are stack-navigated (back = previous step, not back to search).
Progress indicator (step dots) shown in header on all booking screens.
State persists through steps in bookingFlowStore. Cleared on booking confirmation or cancellation.

bookingFlowStore state:
  {
    tripId: string | null
    trip: Trip | null
    selectedPickupLocationId: string | null
    selectedDropoffStopId: string | null
    selectedSeatNumbers: number[]
    passengerDetails: PassengerDetail[]
    pointsToRedeem: number
    discountEtb: number
    totalFareEtb: number
    paymentMethod: PaymentMethod | null
    paymentPhone: string
    bookingId: string | null       // set after POST /bookings
    transactionId: string | null   // set after POST /payments/initiate
  }

AXIOS CLIENT  (src/api/client.ts)
  const apiClient = axios.create({
    baseURL: process.env.EXPO_PUBLIC_API_BASE_URL + '/api/v1',
    timeout: 30000,
    headers: { 'Content-Type': 'application/json' }
  });

  REQUEST INTERCEPTOR:
    1. Read access token from expo-secure-store (ADRASH_ACCESS_TOKEN)
    2. Attach: Authorization: Bearer {token}
    3. Attach: X-Request-ID: crypto.randomUUID()
    4. Attach: X-App-Version: Constants.expoConfig?.version
    5. Attach: Accept-Language: i18n.language

  RESPONSE INTERCEPTOR — 401 (token refresh):
    On 401 AND request._retry !== true:
      Set request._retry = true (prevent infinite loop)
      Read ADRASH_REFRESH_TOKEN from SecureStore
      POST /auth/refresh { refreshToken }
      On success: save new tokens, retry original request
      On failure: clearTokens() → router.replace('/(auth)/phone')
                  Toast: "Your session has expired. Please sign in again."

  RESPONSE INTERCEPTOR — 403 AgreementUpdateRequired:
    if (error.response?.data?.code === 'Auth.AgreementUpdateRequired')
      router.replace('/agreement')

  RESPONSE INTERCEPTOR — network error:
    If no response (offline): dispatch offline event to networkStore
    Show persistent offline banner

SECTION 7 — SECURITY
════════════════════════════════════════════════════════════════════════════════

TOKEN STORAGE
  expo-secure-store: AES-256 hardware-backed on Android (Keystore), iOS Keychain
  Keys: ADRASH_ACCESS_TOKEN, ADRASH_REFRESH_TOKEN
  iOS option: keychainAccessible: SecureStore.WHEN_UNLOCKED_THIS_DEVICE_ONLY
              (prevents cross-device iCloud Keychain restore)
  NEVER store tokens in:
    AsyncStorage, MMKV, Redux persist, or any unencrypted storage
  On logout: SecureStore.deleteItemAsync for both keys
  On uninstall: Android SecureStore auto-cleared. iOS persists — handled by WHEN_UNLOCKED_THIS_DEVICE_ONLY

BIOMETRIC AUTHENTICATION
  expo-local-authentication
  Check: hasHardwareAsync() && isEnrolledAsync() before offering
  Prompt options:
    promptMessage: "Sign in to አድራሽ"
    fallbackLabel: "Use phone number" (iOS) / system fallback (Android)
    disableDeviceFallback: false (allow device PIN as fallback)
  On success: validate session server-side via token refresh
  Biometric does NOT replace server-side token validation — it only unlocks local tokens
  After 3 biometric failures: fall back to OTP flow automatically
  Opt-in: user enables explicitly in Profile. Preference in MMKV (not sensitive).

CERTIFICATE PINNING
  Implement for production builds only (__DEV__ === false)
  Pin SHA-256 fingerprints of api.adrash.et TLS certificate (leaf + intermediate)
  On pin failure: "Connection security error. Please update the app."
  Update pins via EAS OTA update when certificate rotates

JAILBREAK / ROOT DETECTION
  react-native-device-info: isRooted() check on app start
  On detection: show warning (do NOT block — false positives exist)
  Log security event to API if endpoint available
  Do not prevent booking — warn-only policy

SCREENSHOT PREVENTION
  Android: expo-screen-capture (FLAG_SECURE equivalent)
    Apply to: payment screen, ticket QR screen, earnings (if any)
  iOS: Blur overlay when app backgrounds (AppState listener)
    Show blur overlay on QR ticket when backgrounded (protect from multitasking preview)

DEEP LINK SECURITY
  All incoming deep links from AfroSMS or share links:
  Validate scheme: adrash-passenger://
  Validate route: only known routes allowed
  Validate parameters: UUID format for IDs
  Do NOT auto-execute actions from deep links — navigate only, let user confirm

PAYMENT SECURITY
  No payment credentials stored anywhere on device
  Mobile money phone numbers: NOT stored after payment session ends
    Clear paymentPhone from bookingFlowStore after transaction completes
  Payment session timeout: 5 minutes (matches server-side)
  Transaction ID: stored only in bookingFlowStore (MMKV not backed) — cleared after flow
  "No card information is stored" trust signal on payment screen

INPUT VALIDATION (client-side, Zod)
  Phone: Ethiopian format validator before API call
  Passenger names: min 2 chars, max 50 chars, no numbers
  Next of kin phone: same validation as passenger phone
  Points redemption: must be integer ≥ 100, must not exceed balance
  QR payload: non-empty, min length check before API call

SENSITIVE DATA DISPLAY
  Payment method phone: shown on confirmation screen, then cleared from store
  Next-of-kin details: not displayed after booking confirmation
  Booking reference: safe to show — not sensitive

SESSION MANAGEMENT
  Access token: 15-minute expiry, auto-refreshed by interceptor
  Refresh token: 30-day expiry
  After 30 days: forced re-OTP (user sees "session expired" message)
  No idle timeout (passengers may check ticket while travelling)
  Force logout: on 401 from refresh endpoint, on 403 AccountSuspended

CLIPBOARD SECURITY
  Booking reference copy: safe, auto-expires from clipboard after 60 seconds
    Clipboard.setStringAsync(ref)
    setTimeout(() => Clipboard.setStringAsync(''), 60000)
  Payment reference: show on screen only, no copy feature

NETWORK SECURITY
  HTTPS only. Axios configured with no HTTP fallback.
  Timeout: 30 seconds on all requests
  Network monitoring: @react-native-community/netinfo
  Offline state: clear error messaging, no silent failures

PRIVACY
  No analytics SDK in v1 (no Mixpanel, Amplitude, Firebase Analytics)
  Crash reporting via Sentry only
  Sentry config: strip phone numbers, location data, payment references from error events
  Minimum data collection: only what is needed for the feature
  No GPS collection on the passenger app (location is driver-side only)
  When passenger shares trip link: only trip ID shared, not personal data

The passenger app does NOT have background location or critical offline operations

OFFLINE BANNER

  Show "Last updated {time}" indicator when showing cached ticket

CRITICAL ACTIONS — OFFLINE BEHAVIOUR
  Book a trip:       Requires connectivity (creates server record)
  Pay:               Requires connectivity (initiates payment with provider)
  Cancel booking:    Requires connectivity
  Start tracking:    Requires connectivity (SignalR)
  View ticket:       Available offline (from MMKV cache)
  View trip history: Available offline (TanStack Query cache)
  Submit review:     Queue if offline. Flush on reconnect. (MMKV queue)

    Show: "Connection lost — waiting to reconnect"
    On reconnect: resume polling POST /payments/verify
